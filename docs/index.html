Descripción: Secuencia completa desde que el usuario crea un pedido hasta que se notifica.

Observadores: Se ejecutan sincrónico en el mismo request

Patrones de Diseño Implementados

1. Factory Pattern (Servicio de Pagos)

PaymentFactory.obtener_procesador("tarjeta") // ← Dinámico → TarjetaProcessor PaymentFactory.obtener_procesador("paypal") // ← Sin modificar Factory → PayPalProcessor PaymentFactory.obtener_procesador("cripto") // ← Agregar nuevo = Easy → CryptoProcessor

2. Strategy Pattern (Procesadores)

Cada procesador implementa PaymentProcessor con su propia lógica, intercambiables en tiempo de ejecución.

3. Observer Pattern (Servicio de Pedidos)

for observador in service.observadores: observador.actualizar(pedido) // Observadores: // - InventoryObserver: Decrementa stock // - NotificationObserver: Envía email // - LogObserver: Registra evento // - (Agregar nuevo sin tocar OrderService)

4. Repository Pattern (Acceso a Datos)

Abstrae el acceso a base de datos. El servicio no sabe si usa PostgreSQL, MySQL, etc.

5. Event-Driven Architecture (RabbitMQ)

Desacoplamiento completo entre servicios mediante publicación de eventos en message broker.

Conclusión

Arquitectura Lograda:
  • Escalable: Cada servicio escala independientemente
  • Modular: Servicios con responsabilidades claras
  • Extensible: Factory y Observer patterns facilitan nuevas funcionalidades
  • Resiliente: Fallos aislados, message broker persistente
  • Desacoplado: REST entre clientes-servicios, AMQP entre servicios
  • Testeable: Componentes aislados y mockeable

El sistema está diseñado para crecer y escalar con el negocio.
